ZeroTrust Docs

Everything you need to integrate, configure, and extend ZeroTrust — the issue tracker built for engineering teams that care about zero-noise workflows.

Updated Feb 2026
v2.4.0

Quick Start

Get a working ZeroTrust instance running in under five minutes. You'll need Python 3.10+ and a PostgreSQL database (SQLite works fine for local development).

  1. Clone the repository
    Pull the latest stable release from GitHub.
  2. Install dependencies
    All Python packages are pinned in requirements.txt.
  3. Configure environment variables
    Copy .env.example to .env and set your database URL and secret key.
  4. Run migrations & start the server
    Apply the schema and boot the Flask dev server on port 5000.
bash 
git clone https://github.com/mannansainidev/ZeroTrust.git
cd zerotrust
pip install -r requirements.txt
cp .env.example .env
flask db upgrade
flask run
TipThe dev server boots on http://localhost:5000. Your first login will trigger the admin setup wizard.

Authentication

ZeroTrust uses session-based authentication for the web UI and bearer tokens for the REST API. All API requests must include a valid token in the Authorization header.

http 
Authorization: Bearer <your-api-token>
SecurityTokens are scoped per user and expire after 30 days of inactivity. Never commit tokens to source control.

Generating a Token

Navigate to Settings → API Tokens in the web UI, or call the token endpoint directly with your credentials:

POST /api/auth/token 
{
  "email": "you@example.com",
  "password": "••••••••"
}

Issues

Issues are the core entity in ZeroTrust. Each issue belongs to a project, carries a priority level, and transitions between active and closed states.

NoteIssue references (the ref field) are auto-generated in the format FIG-042 and are immutable after creation.

API Endpoints

GET /api/issues List all issues
POST /api/issues Create a new issue
GET /api/issues/:id Retrieve a single issue
PATCH /api/issues/:id Update an issue
DELETE /api/issues/:id Permanently delete an issue

Field Reference

The issue object returned by the API contains the following fields:

FieldTypeReq.Description
id integer required Auto-assigned unique numeric identifier.
ref string required Human-readable reference, e.g. FIG-042.
title string required Short descriptive title. Max 255 characters.
project string required Name of the associated project.
priority enum required One of High, Medium, or Low.
status enum optional active or closed. Defaults to active.
date date optional Creation date (e.g. Dec 5). Defaults to today.
owner_id integer optional Foreign key referencing the assigned user.

Views — List & Grid

The default view is List. Issues render in a sortable table with columns for ref, title, project, priority, date, and owner. Clicking any column header sorts the table client-side.

Switch to Grid for a card-based layout suited to scanning many issues at once. Both views share the same search and filter controls in the toolbar.

URL Parameters

ParamTypeDefaultDescription
viewstringlistlist, grid, board, or timeline
statusstringactiveactive or closed
prioritystring—Filter by High, Medium, or Low
projectstring—Filter by exact project name

Board View

Board groups issues into three columns by priority — High, Medium, and Low. Each column shows a count badge and scrolls independently. Issues can be closed directly from the board without navigating away.

Timeline View

Timeline groups issues by their date field in reverse chronological order. Each group renders a date label followed by a dot-marked list of issues for that day, colour-coded by priority.

Filtering & Search

The toolbar provides two server-side filters (priority, project) and one client-side search input. Search filters rows in real time by matching against the issue title, ref, and project name without a round-trip to the server.

example url 
/dashboard?status=active&view=list&priority=High&project=Backend
Stats stripClicking any card in the stats strip applies the corresponding priority filter and highlights the selected card.

Error Codes

The API returns standard HTTP status codes alongside a JSON body with a machine-readable code and a human-readable message.

StatusCodeDescription
400bad_requestMissing or invalid fields in the request body.
401unauthorizedMissing or expired bearer token.
403forbiddenToken lacks permission for this action.
404not_foundRequested issue or resource does not exist.
422validation_errorData passed validation but failed a business rule.
500server_errorUnexpected server-side error — check application logs.
